# Индустриальное Возрождение — План разработки

> Fabric 1.20.1 + KubeJS + Python Middleware + OpenAI/Anthropic API

---

## Архитектура проекта (напоминалка)

```
Minecraft Server (Custom Mod + KubeJS)
        ↕ WebSocket
Python Middleware (FastAPI)
        ↕ HTTP
LLM/VLM (Claude / GPT-4o)
```

---

## Фаза 1 — Система зданий и прогрессия

> Цель: игрок строит здания, игра их проверяет и выдаёт прогрессию.

### 1.1 Инфраструктура валидации ✅
- [x] Кастомный мод: сканер региона (список блоков → JSON) — `command/ScanCommand.java`
- [x] Кастомный мод: снимок экрана с 3 ракурсов по команде — рендер 3 видов (top-down, iso NE, iso SW) через PIL
- [x] Middleware: изометрический рендер блок-дампа через PIL — `vlm/renderer.py` (map_color + keyword fallback)
- [x] WebSocket канал: Minecraft ↔ Middleware — `network/MiddlewareClient.java` + `app/routes.py`

### 1.2 KubeJS — техническая валидация ✅
- [x] Триггер: игрок активирует "проверку здания" (предмет/команда) — `/validatebuilding <type> [radius]`
- [x] Скрипт: проверка замкнутости периметра стен — `checks/perimeter_check.js` (flood-fill)
- [x] Скрипт: проверка наличия обязательных блоков по типу здания — `checks/required_blocks_check.js` (22 типа зданий)
- [x] Скрипт: проверка генерации SU (для инженерных построек) — `checks/su_check.js`
- [x] Скрипт: выдача результата обратно в Middleware — через `/scan` с JSON-меткой

### 1.3 VLM — визуальная валидация ✅
- [x] Промпт-шаблоны для каждого типа здания — `vlm/prompts.py` (все 22 типа)
- [x] Парсинг structured output от GPT-4o (JSON с оценкой 0–100 и комментарием) — `vlm/client.py`
- [x] Кеширование результатов по хешу блок-дампа — in-memory кеш в `vlm/client.py`
- [x] Логика: VLM оценивает субъективное, KubeJS — бинарное
- [x] Изометрический рендер блок-дампа через Pillow — `vlm/renderer.py`

### 1.4 Агрегатор валидации ✅
- [x] Middleware: объединение результатов KubeJS + VLM — `aggregator.py` (ValidationReport)
- [x] Логика двухфазной валидации (KubeJS мгновенно, VLM async с прогресс-баром) — `validate_progress` → `validate_result`
- [x] Feedback игроку: что именно не так, если провалено — `summary_lines` + текстовый прогресс-бар → KubeJS через `/civresult`
- [x] Обработка "свободных построек" (VLM без чеклиста → атрибуция → малый бонус) — команда `/freebuild`, бонус XP

### 1.5 Система зданий
- [ ] JSON реестр всех зданий с требованиями и наградами
- [ ] Здания Мастера продовольствия (колодец, ферма, амбар, пекарня, таверна, теплица)
- [ ] Здания Мастера инженерии (мельница, кузница, прессовочный цех, сборочная линия, депо)
- [ ] Здания Мастера обороны (башни Тир 1–4, казарма, оружейная, застава)
- [ ] Здания Мастера магии (библиотека, алхимлаб, башня мага, обсерватория, архив)
- [ ] Здания Мэра (ратуша, банк, рыночная площадь)

### 1.6 Прогрессия предметов
- [ ] KubeJS: блокировка/разблокировка рецептов в зависимости от построенных зданий
- [ ] KubeJS: блокировка технологии при гибели профильного NPC
- [ ] CityState (SQLite): хранение статуса всех зданий, тира города
- [ ] Тир города считается автоматически по набору построенных зданий

### 1.7 Карта города (in-game GUI)
- [ ] Кастомный Screen: рендер terrain из chunk heightmap
- [ ] Слой зданий: иконки по координатам из CityState
- [ ] Network packets: broadcast при регистрации нового здания всем клиентам
- [ ] Клик по зданию: карточка с мета-информацией (тир, мастер, что разблокировано)
- [ ] Якорь карты: координаты центра фиксируются при основании поселения

## Фаза 2 — Система NPC

> Цель: Совет из 5 Мастеров управляет городом, рядовые NPC работают по скриптам.

### 2.1 Инфраструктура NPC
- [ ] NPC Manager в Middleware: профили Мастеров в JSON (профессия, память, отношение)
- [ ] Глобальный CityState инжектируется в каждый запрос к LLM
- [ ] Structured output формат: `{dialog, action, payload, memory_update}`
- [ ] Очередь диалогов: один Мастер = одна очередь (нельзя раздвоиться)

### 2.2 Совет города (LLM-driven)
- [ ] 👑 Мэр: квесты, налоги, глобальные решения, GUI Ратуши
- [ ] ⚙️ Мастер инженерии: запуск валидации зданий, GUI производства
- [ ] 🌾 Мастер продовольствия: управление фермами, население, GUI склада
- [ ] ⚔️ Мастер обороны: башни, стражники, GUI тактической карты
- [ ] 🔮 Мастер магии: технологии, чертежи, GUI древа знаний
- [ ] Инициативные события: Мастер сам инициирует квест при изменении CityState

### 2.3 Память NPC
- [ ] RAG-лайт: память как список фактов, топ-5 релевантных по косинусному расстоянию
- [ ] Обновление памяти после каждого диалога
- [ ] Смерть Мастера: технология блокируется до появления нового поселенца

### 2.4 Рядовые NPC (scripted)
- [ ] Шаблонные диалоги по профессии (фермер, кузнец, стражник)
- [ ] Логика торговли без LLM
- [ ] Подчинение приказам своего Мастера

### 2.5 Поселенцы
- [ ] Лоровый механизм прибытия: поезд/повозка из далёкой цивилизации
- [ ] Система найма за валюту через GUI Мэра
- [ ] Профессии поселенцев влияют на доступные роли

### 2.6 GUI Мастеров
- [ ] Базовый шаблон кастомного Screen (Fabric)
- [ ] GUI Мастера инженерии: запуск валидации, история построек
- [ ] GUI Мастера обороны: тактическая карта, расстановка стражи
- [ ] GUI Мэра: статус города, найм, управление
- [ ] GUI Мастера продовольствия: ресурсы, население
- [ ] GUI Мастера магии: древо технологий, чертежи

## Фаза 3 — Система рейдов

> Цель: Государство злых атакует поселение, используя карту защищённости.

### 3.1 Коэффициент защищённости
- [ ] Валидация башен (Тир 1–4): KubeJS + VLM по отдельным чеклистам
- [ ] Алгоритм покрытия периметра: дуги между башнями, расчёт брешей
- [ ] Отображение брешей на карте города (красные зоны)
- [ ] Интеграция с CityState: защищённость обновляется при постройке/разрушении башни

### 3.2 Планировщик рейдов
- [ ] Middleware: Raid Planner читает CityState и карту защищённости
- [ ] LLM генерирует план рейда: волны, типы врагов, целевые бреши
- [ ] Эволюция врагов: тир города → сложность рейда
- [ ] Расписание рейдов: Enhanced Celestials триггер → Middleware → план → KubeJS spawn

### 3.3 Умные враги
- [ ] Враги атакуют бреши, а не случайные точки
- [ ] Разные типы: разведчики, саперы, осадные машины (по тиру города)
- [ ] Improved Mobs: разрушение слабых блоков, лестницы через стены

### 3.4 Активная оборона
- [ ] Стражники-NPC по приказу Мастера обороны занимают башни
- [ ] Create Big Cannons: интеграция с системой башен Тир 3–4
- [ ] Потери в рейде: убыль населения, блокировка технологий при гибели Мастера
